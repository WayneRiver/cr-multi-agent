# 代码评审多智能体系统 - 完整工作流程(是用来了解项目的)

## 阶段0：输入接收

系统接收以下输入：
- 代码内容（完整文件或diff）
- 编程语言（Python、Java、Go、JavaScript、TypeScript）
- PR描述（可选，用于Planner决策）
- commit哈希（可选，用于缓存）

触发方式可以是：GitHub Webhook、CLI命令、或FastAPI手动调用。


## 阶段1：缓存检查

如果输入包含commit哈希，系统先查询Redis。

缓存key格式：cr:result:{commit_hash}

如果缓存命中，直接返回上次的评审结果，流程结束。

如果缓存未命中，继续进入下一阶段。


## 阶段2：Planner决策

Planner负责决定启用哪些Agent。

决策分两层：

第一层：规则判断（80%的场景，不调用LLM）
- 如果语言是Python或Java → 启用全部4个Agent（规范员、安全员、逻辑员、可读员）
- 如果语言是Go → 启用3个Agent（跳过安全员）
- 如果语言是JavaScript或TypeScript → 启用3个Agent（跳过逻辑员）
- 如果PR描述包含“格式化”或“注释修改” → 只启用规范员和可读员

第二层：LLM判断（20%的边界场景，调用一次LLM）
触发条件：
- PR描述包含“数据库”或“SQL” → 强制启用安全员
- PR描述包含“并发”或“多线程” → 强制启用逻辑员
- 代码变更超过20个文件 → 考虑降级，减少Agent数量

Planner输出：enabled_agents列表


## 阶段3：提前终止判断

系统检查规范员是否在enabled_agents中。

如果在，先快速运行规范员的格式检查。

如果规范员发现严重格式问题超过3个：
- 直接返回“代码格式问题严重，请先修复”
- 跳过所有后续Agent
- 流程结束

如果格式问题少于3个，继续进入下一阶段。


## 阶段4：主流程执行（半并行）

主流程分三层执行：

第一层（并行执行）：
- 规范员 Agent 和 安全员 Agent 同时运行
- 两个Agent互不依赖，可以并行
- 规范员通过MCP调用Ruff工具进行格式检查
- 安全员直接用LLM分析安全漏洞
- 这一层设置10秒超时

第二层（串行执行）：
- 逻辑员 Agent 在第一层完成后启动
- 逻辑员的输入包括：原始代码 + 安全员的发现
- 逻辑员重点分析安全员标记的高风险代码区域
- 这一层设置15秒超时（因为需要更复杂的推理）

第三层（串行执行）：
- 可读员 Agent 在第二层完成后启动
- 可读员的输入包括：原始代码 + 规范员发现 + 安全员发现 + 逻辑员发现
- 可读员综合所有前期发现，给出整体可读性建议
- 这一层设置10秒超时

每一层内部，每个Agent都遵循ReAct循环：思考当前需要检查什么、执行检查、观察结果、输出发现。


## 阶段5：失败检测与降级

系统监控每个Agent的执行状态。

定义失败的情况：
- LLM调用超过超时时间
- 返回格式不符合Finding Schema
- 返回空结果

降级策略分三级：

降级一（单个Agent失败）：
- 系统自动重启失败的Agent
- 最多重试1次
- 如果成功，继续正常流程

降级二（部分Agent失败）：
- 如果重试后仍有1-2个Agent失败
- 系统跳过失败的Agent
- 只使用成功Agent的结果继续
- 在最终输出中增加提示：“部分检查不可用”

降级三（大面积失败）：
- 如果成功Agent少于2个
- 系统切换到单Agent兜底模式
- 使用逻辑员作为全科医生，重新分析全部代码
- 在最终输出中增加警告：“完整评审不可用，已降级到单Agent模式”


## 阶段6：结果汇总

Aggregator收集所有成功Agent的输出。

汇总步骤：
1. 合并所有Finding
2. 去重：相同位置、相同类型的问题只保留一条
3. 冲突解决：如果不同Agent对同一段代码有不同判断，取severity更高的
4. 排序：按severity从critical到info排序

Aggregator输出统一的Finding列表。


## 阶段7：缓存写入与输出返回

系统将本次评审结果写入Redis：
- key: cr:result:{commit_hash}
- value: 完整的评审结果
- 过期时间：7天

然后返回结构化输出，包含：
- 状态（通过/驳回/部分通过）
- 所有Finding
- 被跳过的Agent列表
- 是否触发了降级
- 总执行时间


## 阶段8：可选的后处理

如果系统配置了GitHub集成：
- 将Finding转换为GitHub PR评论格式
- 自动发布到对应的PR页面

如果系统配置了Streamlit面板：
- 实时更新每个Agent的执行状态
- 显示每个Agent的中间输出
- 展示降级事件日志
